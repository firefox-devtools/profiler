pq: Usage Guide

pq (Profile Querier) queries Firefox performance profiles loaded into a persistent
daemon. Load a profile once, then query it interactively without reloading.


SESSION MODEL

  pq uses a daemon/session model:
    1. Load a profile with "pq load" -- starts a background daemon
    2. Query the loaded profile with any other commands
    3. Stop with "pq stop" when done

  The daemon persists your selected thread, zoom ranges, and filter stacks across queries.

  For automation or concurrent work, prefer an explicit session ID so each workflow
  gets isolated state:
    pq load profile.json.gz --session run-a
    pq profile info --session run-a
    pq thread select t-0 --session run-a
    pq thread samples --session run-a


CORE WORKFLOW

  Step 1: Load a profile
    pq load profile.json.gz
    pq load profile.json.gz --session run-a   Recommended for automation or concurrent work

  Step 2: Explore the profile
    pq profile info          Overview: processes, threads, time range, CPU activity
    pq profile info --all    Show all processes and threads (not just top 5)
    pq profile info --search GeckoMain   Filter by process/thread name, pid, or tid

  Step 3: Select a thread to analyze (requires Step 2 -- profile info builds thread handles)
    pq thread select t-0     Select a specific thread (persists for future commands)

  Step 4: Analyze CPU usage
    pq thread samples                             Hot functions list (self time, sorted by impact)
    pq thread samples-top-down                    Top-down call tree (where does CPU time originate)
    pq thread samples-bottom-up                   Bottom-up call tree (what calls the hot functions)
    pq thread functions                           All functions with self/total CPU percentages

    All samples commands exclude idle by default so percentages reflect active CPU time.
    Use --include-idle to include idle samples (e.g. to see what fraction of wall time is idle).

    Use --search to focus the call tree on paths containing a specific function:
    pq thread samples-top-down --search GC
    pq thread samples-bottom-up --search "JS::Compile"
    pq thread samples --search malloc

  Step 5: Analyze markers (browser events, timers, etc.)
    pq thread markers                          All markers with aggregated stats
    pq thread markers --category Layout        Filter by category
    pq thread markers --search DOMEvent        Filter by name substring
    pq thread markers --min-duration 10        Only markers >= 10ms
    pq thread markers --has-stack              Only markers with stack traces

  Step 6: Drill into specifics
    pq marker info m-1234     Full details for a marker (from handles in marker list)
    pq marker stack m-1234    Full stack trace at the time of a marker
    pq function info f-12     Function details (source location, library)
    pq function expand f-12   Show full untruncated function name

  Step 7: Filter to focus the analysis (see FILTERS section below)

    Ephemeral -- applies only to that one command, sticky state unchanged:
    pq thread samples --excludes-function f-184     Ignore one noisy function
    pq thread samples --root-at f-500               Show only work inside f-500
    pq thread samples --during-marker --search Paint  Only samples during Paint markers

    Sticky -- persists across all subsequent commands until popped:
    pq filter push --merge f-142,f-143             Merge allocators out of stacks
    pq filter push --includes-function f-500       Keep only samples containing f-500
    pq thread samples                              Uses sticky filters automatically
    pq filter pop                                  Remove last filter
    pq filter clear                                Remove all filters


HANDLE SYSTEM

  Handles are short IDs shown in command output that reference specific items:
    t-0, t-1   Thread handles (from "profile info")
    m-1234     Marker handles (from "thread markers")
    f-12       Function handles (from "thread samples", "thread functions")
    ts-6       Timestamp handles (named points in time, usable with "zoom push")

  Thread and marker handles are stable within a session.
  Function handles (f-N) are stable across sessions for the same profile --
  they are direct indices into the shared function table.


ZOOM: FOCUS ON A TIME RANGE

  Zoom restricts all subsequent queries to a specific time window. Useful for
  drilling into a specific jank period, page load phase, or marker duration.

    pq zoom push 2.7,3.1       Zoom to 2.7s-3.1s (relative to profile start)
    pq zoom push m-158         Zoom to the time range of marker m-158
    pq zoom push ts-6,ts-12    Zoom to the range between two named timestamps
    pq zoom pop                Undo the last zoom
    pq zoom clear              Return to full profile view

  After zooming, all thread/marker/sample commands automatically apply the filter.
  Use "pq status" to confirm the active zoom stack (along with thread and filters).


FILTERS

  Filters narrow which samples appear in analysis results. They work in two modes:

  EPHEMERAL (one command only)
    Add filter flags directly to thread samples/functions commands. They apply
    only to that invocation; the sticky filter stack is unchanged. Multiple flags
    may be combined on one command and are applied left to right.

      pq thread samples --excludes-function f-184
      pq thread samples --merge f-142,f-143 --root-at f-500 --limit 30
      pq thread functions --includes-function f-500

  STICKY (persists across commands)
    Use "pq filter push" to add a filter to the current thread's stack.
    Every subsequent analysis command sees all pushed filters automatically.
    Filters are per-thread: each thread has its own independent filter stack.
    Exactly one filter flag per "filter push" command.

      pq filter push --merge f-142,f-143         # add filter 1
      pq filter push --includes-function f-500   # add filter 2 (AND: both must pass)
      pq thread samples                          # both filters active
      pq thread functions                        # both filters active
      pq filter pop                              # remove filter 2
      pq filter pop 2                            # remove last 2 filters
      pq filter list                             # show active filters in push order
      pq filter clear                            # remove all filters

  Use "pq status" to inspect the full session state: selected thread, zoom stack, and
  all per-thread filters.

  Sticky + ephemeral compose: sticky filters apply first (already in the transform
  stack), then ephemeral filters layer on top for that one invocation only.

  AVAILABLE FILTER FLAGS

    Inclusion -- keep only samples whose stack matches:
      --includes-function f-N,...       stack contains any of these funcs (OR)
      --includes-any-function f-N,...   alias for --includes-function
      --includes-prefix f-N,...         stack starts with this root-first sequence
      --includes-suffix f-N             leaf (innermost) frame is f-N
      --during-marker --search <text>   sample timestamp inside a matching marker
      --outside-marker --search <text>  sample timestamp outside all matching markers

    Exclusion -- drop matching samples:
      --excludes-function f-N,...       stack contains any of these funcs
      --excludes-any-function f-N,...   alias for --excludes-function

    Stack transforms -- modify stack structure:
      --merge f-N,...                   remove funcs from stacks (A→f-N→B becomes A→B)
      --root-at f-N                     re-root all stacks at f-N (subtree within f-N)

  COMBINING FILTERS

    OR within one push: --includes-function f-1,f-2 keeps samples with f-1 OR f-2.
    AND across pushes: two separate "filter push" calls both must pass.
    Push order matters: each filter sees the stack as left by the prior filter.

    Example -- only samples containing f-500, during Paint markers, after merging noise:
      pq filter push --merge f-142,f-143
      pq filter push --includes-function f-500
      pq filter push --during-marker --search Paint


JSON OUTPUT

  Add --json to any command to get structured JSON output, useful for piping to jq:

    pq thread samples --json | jq '.rows[0]'
    pq thread markers --json | jq '[.rows[] | select(.totalDuration > 50)]'
    pq profile info --json | jq '.threads | length'


COMMON ANALYSIS PATTERNS

  Find what's consuming CPU:
    pq thread info                          Pick the main thread (usually GeckoMain)
    pq thread select t-0
    pq thread samples                       Top hot functions by self time (active only)
    pq thread samples-top-down             See the full call tree
    pq thread samples-top-down --search GC  Focus tree on paths through "GC"

  Investigate a jank/slow period:
    pq thread markers --min-duration 50    Find long-running markers (>50ms = jank)
    pq zoom push m-158                     Zoom into that marker's time range
    pq thread samples                      What was executing during that period?
    pq zoom pop                            Back to full profile

  Eliminate allocator noise from a call tree:
    pq thread functions --search malloc    Find allocator function handles (e.g. f-142)
    pq thread samples --merge f-142        Try ephemerally first
    pq filter push --merge f-142,f-143     Make it sticky for all subsequent commands
    pq thread samples                      Clean call tree, filters persist
    pq filter clear

  Focus on work inside a specific function:
    pq thread functions --search PresentImpl   Note the handle (e.g. f-500)
    pq thread samples --root-at f-500          Ephemeral: subtree rooted at f-500
    pq filter push --includes-function f-500   Sticky: only samples containing f-500
    pq filter push --root-at f-500             Sticky: re-root at f-500
    pq thread samples-top-down                 Call tree within f-500 only

  Correlate CPU with a specific event type:
    pq thread markers --search Paint           Check Paint marker frequency/duration
    pq filter push --during-marker --search Paint
    pq thread samples                          What runs during Paint?
    pq thread functions                        Which functions are active during Paint?
    pq filter clear

  Find slow layout or script execution:
    pq thread markers --category Layout --min-duration 5
    pq thread markers --category JavaScript --min-duration 10
    pq thread markers --search Reflow --min-duration 5

  Deep dive on a specific function:
    pq thread samples                   Find hot function, note its handle (f-12)
    pq function info f-12               See callers, callees, source location
    pq function expand f-12             See full name if truncated

  Group markers by event type:
    pq thread markers --search DOMEvent --group-by field:eventType
    pq thread markers --auto-group
    pq thread markers --group-by type,name

  Analyze network activity:
    pq thread markers --category Network
    pq thread markers --category Network --min-duration 100
    pq thread markers --search "http" --category Network


UNDERSTANDING THE OUTPUT

  Self time vs total time:
    Self time   Samples where this function was the innermost frame (actually executing)
    Total time  All samples where this function appeared anywhere in the stack

    Self time is more actionable -- it pinpoints where CPU time is actually spent.

  Samples:
    Profiles are sampled at regular intervals (typically 1ms). Each sample is a snapshot
    of the call stack. Higher sample counts = more time spent. Counts are relative
    within a profile.

    By default, samples commands drop idle samples before computing percentages, so
    percentages reflect how the thread spent its active CPU time. Pass --include-idle
    to include idle samples and see percentages relative to wall time instead.

  Call tree views:
    samples-top-down    Root frames at top, drilling down to leaves.
                        Use to understand the calling structure and where time originates.
    samples-bottom-up   Hot leaf functions at top with their callers.
                        Use to understand what calls a frequently-seen function.

  Scoring strategies (controls node ranking when tree is truncated with --max-lines):
    exponential-0.9 (default)  Balances depth vs breadth
    percentage-only            Rank purely by CPU percentage (flatter output)
    harmonic-1.0               Emphasizes deeper paths in the tree


TIPS

  - Start with "profile info" to understand what's in the profile; use --all to see every
    process and thread, or --search <term> to filter by name, pid, or tid
  - Select a thread before querying -- GeckoMain is usually the main thread
  - Self time is more actionable: focus on high self-time functions for optimization
  - Samples percentages are relative to active (non-idle) time by default. Use --include-idle
    to see wall-time percentages. Idle time is not a finding on its own -- only investigate it
    during a period the thread should be busy (e.g. inside a zoom), which may indicate lock
    contention or blocking on another thread.
  - Use --search to narrow the call tree to branches you care about: --search GC shows only
    call paths that pass through a GC-related function
  - Zoom into slow periods: use "zoom push m-<handle>" to focus on a slow marker
  - Try filters ephemerally first (as flags on thread commands) before committing with "filter push"
  - "filter push --merge" is great for hiding allocator/profiler noise without losing context
  - "filter push --root-at f-N" + "filter push --includes-function f-N" tightly focuses
    analysis on one function's work
  - "filter push --during-marker --search X" is powerful for correlating CPU work with events
  - Sticky filters compose: each push narrows further (AND semantics across pushes)
  - "filter list" shows filters in push order; "filter pop" removes from the end
  - Use --json when you need to process or filter results programmatically
  - Marker handles are stable within a session; function handles (f-N) are stable across sessions
  - Check "status": run "pq status" to confirm selected thread, active zoom, and active filters


SESSION MANAGEMENT

    pq session list                               List all running daemon sessions (* marks current)
    pq session use <id>                           Switch the current session
    pq stop                                       Stop the current session
    pq stop <id>                                  Stop a specific session
    pq stop --all                                 Stop all sessions
    pq load profile.json.gz --session my-session  Named session
    pq thread info --session my-session           Query a specific session


ERROR HANDLING

  "No running session"  Run "pq load <profile>" first
  "Thread not found"    Run "pq profile info" to see valid thread handles
  "Marker not found"    Handles are session-scoped; reload if session restarted
